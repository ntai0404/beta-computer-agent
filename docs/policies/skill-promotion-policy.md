# Skill Promotion Policy

This document defines the lifecycle and promotion criteria for skills in Beta.

---

## Skill Lifecycle

```
candidate
 └─ evaluating
      ├─ approved
      │    └─ active
      │         └─ deprecated
      │               └─ archived
      └─ rejected
            └─ archived
```

---

## States

| State | Description |
|---|---|
| `candidate` | Newly proposed skill; not yet evaluated |
| `evaluating` | Under functional, safety, and repeatability assessment |
| `approved` | Passed all evaluations; ready for activation |
| `active` | Available for agents to invoke |
| `deprecated` | Still active but scheduled for retirement; not recommended for new tasks |
| `rejected` | Failed evaluation; may be archived or revised |
| `archived` | No longer active; retained for history and audit |

---

## Candidate Creation

Any new skill must start in `candidate` state and be placed in `skills/candidates/`.

Agents and the Learning module may propose new skills.
Agents and the Learning module may **not** directly create `active` skills.

---

## Evaluation Requirements

Before a skill may be promoted from `evaluating` to `approved`, it must pass:

### 1. Functional Evaluation

- The skill achieves its stated goal in a controlled test scenario.
- Inputs and outputs match the declared contract.
- Edge cases are handled gracefully.

### 2. Safety Evaluation

- The skill does not bypass the safety policy.
- The skill does not acquire permissions beyond its declared scope.
- The skill does not produce blocked actions.
- The safety risk classification is correct and documented.

### 3. Repeatability

- The skill produces consistent results across multiple test runs.
- Non-deterministic steps are identified and bounded.

### 4. Regression

- The skill does not break existing active skills or known task workflows.
- Regression test suite must pass.

### 5. Verifier Review

- The Verifier Agent independently evaluates the skill's outputs in test scenarios.
- The Verifier Agent must produce a pass verdict before promotion.

### 6. Human Approval

- A human reviewer must approve the skill before it becomes `active`.
- Approval is recorded in the event log.

---

## Promotion Flow

```
New skill proposed
 └─ placed in skills/candidates/
      └─ functional evaluation
           └─ safety evaluation
                └─ repeatability check
                     └─ regression check
                          └─ verifier review
                               └─ human approval
                                    └─ promoted to skills/active/ (status: active)
                                         or
                                         rejected → skills/rejected/ (status: rejected)
```

---

## Deprecation and Archival

A skill may be deprecated when:

- a better replacement exists;
- the underlying capability has changed and the skill is no longer reliable;
- the skill is rarely or never used.

Deprecated skills remain callable but are flagged as not recommended.
After a defined deprecation period, they are archived.

Archived skills are not callable but are retained for audit and reference.

---

## Self-Generated Skills

Skills generated by the Learning module through automated skill synthesis:

- Always start in `candidate`.
- Must go through the full evaluation flow.
- May not be promoted to `active` without human approval.
- The synthesis rationale must be documented as part of the candidate record.
